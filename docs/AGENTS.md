# AGENTS — описание агентов

Система состоит из маршрутизатора (Supervisor), семи профильных агентов и двух
агентов самопроверки (Reflection, Verifier). Профильные агенты в LLM-режиме
работают через общий agent loop: получают свой system-промпт и JSON-схемы только
своих инструментов. В rule-based режиме их роль выполняют детерминированные
обработчики (`agents/fallback.py`).

Сопоставление «агент → инструменты» задаётся в `app/agents/definitions.py`.

---

## Supervisor (Orchestrator)

- **Роль:** маршрутизатор и сборщик ответа.
- **Ответственность:** понять запрос, выбрать одного профильного агента,
  запустить его, затем Reflection и Verifier, собрать итоговый `AgentMessage`.
- **Инструменты:** нет собственных (делегирует).
- **System prompt:** просит вернуть ровно один код агента из списка
  `task, calendar, planning, prioritization, memory, digest, sleep_fairy`.
- **Ограничения:** при недоступности LLM или невалидном ответе использует
  keyword-роутер; не выполняет действий сам.

## Task Agent

- **Роль:** задачи и чеклисты (стиль Jira).
- **Ответственность:** создавать/изменять/удалять задачи, менять статус и
  приоритет, ставить дедлайн, разбивать на чеклист, отмечать пункты, прогресс.
- **Инструменты:** `create_task`, `update_task`, `delete_task`, `get_task`,
  `list_tasks`, `search_tasks`, `set_task_status`, `set_task_priority`,
  `create_checklist`, `add_checklist_item`, `update_checklist_item`,
  `complete_checklist_item`, `get_task_progress`.
- **System prompt (суть):** статусы `backlog/todo/in_progress/blocked/done/
  archived`; «разбей» → создать задачу и чеклист.
- **Ограничения:** удаление флагуется Reflection как требующее подтверждения.

## Calendar Agent

- **Роль:** локальный календарь (`data/events.json`).
- **Ответственность:** создавать/менять/удалять события, показывать день/неделю,
  искать свободные слоты, проверять конфликты.
- **Инструменты:** `create_event`, `update_event`, `delete_event`,
  `list_events`, `get_day_schedule`, `get_week_schedule`, `find_free_slots`,
  `check_time_conflicts`.
- **Ограничения:** перед созданием события стоит проверить пересечения;
  конфликты сообщаются пользователю.

## Planning Agent

- **Роль:** свободное планирование целей по дням.
- **Ответственность:** распознать цели → разбить на задачи → оценить
  длительность → найти свободные окна → распределить → вернуть план.
- **Инструменты:** `split_goal_into_tasks`, `estimate_task_duration`,
  `schedule_tasks_to_free_slots`, `create_task`, `list_tasks`, `get_task`,
  `list_events`, `find_free_slots`, `get_week_schedule`, `search_memory`,
  `create_checklist`.
- **Ограничения:** учитывает предпочтения из памяти (напр. «утренний человек»);
  если окно не найдено — возвращает предупреждение.

## Prioritization Agent

- **Роль:** ответ на «Что мне делать дальше?».
- **Ответственность:** оценить задачи по дедлайну/важности/срочности/статусу/
  длительности с учётом памяти; дать одну рекомендацию + 2–3 альтернативы +
  объяснение + риски; при необходимости перевести задачу в `in_progress`.
- **Инструменты:** `list_tasks`, `get_task`, `get_task_progress`,
  `get_day_schedule`, `list_events`, `search_memory`, `set_task_status`.
- **Ограничения:** для большой задачи предлагает ближайший маленький шаг.
  Базовая логика — `app/scoring.py`.

## Memory Agent

- **Роль:** личная память пользователя (`data/memory.json`).
- **Ответственность:** сохранять/искать/обновлять/удалять факты, показывать список.
- **Инструменты:** `save_memory`, `search_memory`, `update_memory`,
  `delete_memory`, `list_memory`.
- **Типы:** `user_preference`, `recurring_routine`, `personal_context`,
  `project_context`, `contact`, `rule`, `constraint`.

## Digest Agent

- **Роль:** утренний и вечерний дайджесты.
- **Ответственность:** собрать сводку (задачи, события, просрочки, рекомендация,
  план) и сохранить в `data/digests.json`.
- **Инструменты:** `generate_morning_digest`, `generate_evening_digest`,
  `save_digest`, `list_digests`, `list_tasks`, `list_events`.
- **Ограничения:** `generate_*` сохраняют дайджест автоматически.

## Sleep Fairy Agent («Сонная фея»)

- **Роль:** режим сна.
- **Ответственность:** подвести итог дня (вечерний дайджест), перенести
  незавершённое на завтра, сформировать мягкий план утра, сообщить, что
  критичного не осталось, предложить ритуал завершения дня.
- **Инструменты:** `generate_evening_digest`, `reschedule_overdue_tasks`,
  `list_tasks`, `set_task_status`, `create_event`, `search_memory`, `save_digest`.
- **Ограничения:** спокойный, поддерживающий тон.

## Reflection Agent (детерминированный)

- **Роль:** проверка хода и полноты работы.
- **Проверяет:** выполнена ли цель (вызваны ли инструменты при action-запросе),
  ошибки инструментов, конфликты календаря, задачи без дедлайна (для планирования),
  необходимость подтверждения (необратимые действия), противоречия с памятью.
- **Возвращает:** `ReflectionResult{passed, issues, suggested_fixes,
  requires_user_confirmation, notes}`.

## Verifier Agent (детерминированный)

- **Роль:** проверка фактических изменений данных (в отличие от Reflection).
- **Проверяет в storage:** задача реально создана/обновлена/удалена; событие
  создано/удалено; чеклист создан и связан с задачей; дайджест сохранён; память
  обновлена; задачи/события из планирования реально записаны.
- **Возвращает:** `VerificationResult{passed, checks[], summary}`.

## Safety-net (детерминированный)

- **Роль:** гарантировать выполнение действия, если LLM его не довёл до конца.
- **Зачем:** малые модели (напр. Qwen 4B) часто «анализируют, но не создают» —
  пропускают финальный мутирующий вызов. Safety-net (`app/agents/safety_net.py`)
  после LLM-агента проверяет и при необходимости доделывает шаг детерминированно:
  создаёт чеклист, событие, расписание или запись памяти. Без дублей, если
  действие уже выполнено.
- **Принцип:** LLM остаётся ведущим (диалог + вызовы инструментов), код лишь
  страхует результат. Запускается только в LLM-режиме (в rule-based fallback
  делает всё сам).
